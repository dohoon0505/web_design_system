00·INTRODUCTION

디자인시스템이란 무엇인가

UIUX-DH는 컴포넌트 묶음 이상입니다. 디자이너가 그린 도면, 개발자가 짠 코드, PM이 합의한 정책, 팀이 함께 지키는 운영 프로세스 — 이 네 가지가 하나로 돌아가는 생산 체계입니다.

01 / 규칙

Rules & Policies

토큰 정의, 상태 규칙, 예외 처리 기준. "색은 Hex가 아니라 용도로 부른다"는 선언이 여기 속합니다.

02 / 컴포넌트

Component Library

재사용 가능한 UI 단위. 디자인 토큰을 소비하고, 정책을 구현합니다. 이름은 모든 팀이 하나로 부릅니다.

03 / 연동

Design ↔ Code

Figma의 토큰과 코드의 토큰이 같은 이름으로 살아있어야 합니다. 수동 PR이 아닌 자동 동기화가 지향점입니다.

04 / 운영

Release & Governance

버전, 검증, 피드백 루프. 시스템은 만드는 것이 아니라 굴리는 것입니다. Semantic Versioning, Changelog, Deprecation 절차를 따릅니다.

Note 이 문서가 곧 단일 진실의 출처(Single Source of Truth)입니다. 모든 Figma 라이브러리, 코드 패키지, 운영 문서는 이 정의를 기준으로 동기화됩니다.
01·PRINCIPLES

여섯 가지 원칙

결정이 필요할 때마다 이 원칙으로 돌아옵니다. 토큰을 추가할 때, 컴포넌트를 쪼갤 때, 이름을 붙일 때. 개인의 취향이 아닌 합의된 잣대로 논의하기 위한 기준점입니다.

PRINCIPLE 01

의미로 부른다, 값으로 부르지 않는다.

컬러는 #FFFFFF가 아니라 background-default입니다. Hex는 언젠가 바뀌지만 의미는 바뀌지 않습니다.

"White Gray 100" 같은 값 기반 네이밍 금지
PRINCIPLE 02

라이트와 다크는 같은 이름을 공유한다.

다크는 라이트의 반전이 아니라, 같은 의미 토큰이 다른 값으로 매핑된 결과입니다. content-primary는 양쪽 모두에 존재합니다.

다크 모드만을 위한 별도 토큰 네임스페이스 금지
PRINCIPLE 03

축약하지 않는다.

BTN이 아니라 Button. NAV가 아니라 Navigation. 팀 간 커뮤니케이션의 비용은 축약이 아니라 명확성이 줄입니다.

SVG·CTA 같은 고유명사를 제외하고 약어 금지
PRINCIPLE 04

예외는 시스템 안에서 해결한다.

토큰으로 해결되지 않는 상황이 나타나면, 그것은 새로운 토큰이 필요하다는 신호입니다. 인라인 스타일로 덮어쓰지 않습니다.

style="color: #333" 같은 하드코딩 금지
PRINCIPLE 05

모션은 의미다.

애니메이션은 장식이 아니라 피드백입니다. 상태 변화, 진입/이탈, 성공 같은 순간에만 사용합니다. 짧고, 부드럽고, 예측 가능하게.

장식적 용도의 자동 재생 애니메이션 금지
PRINCIPLE 06

접근성은 기본값이다.

WCAG 2.2 AA를 만족하지 않는 토큰은 존재할 수 없습니다. 텍스트는 4.5:1, UI 요소는 3:1의 대비를 기본으로 갖습니다.

"디자인상 예쁘니까"를 이유로 한 대비 완화 금지
02·TOKEN ARCHITECTURE

세 개의 층.
의미의 계단.

UIUX-DH의 모든 토큰은 세 층으로 구성됩니다. 아래 층은 원시값, 가운데 층은 의미, 위 층은 컴포넌트 전용 맥락. 변경은 위에서 아래로 내려오지 않고, 아래에서 위로 흘러갑니다.

TIER 01 — FOUNDATION

원시 토큰

Primitive Tokens

팔레트의 원재료. UI에 직접 사용하지 않습니다. 디자이너만 접근합니다.

--p-indigo-500: #4F46E5;
--p-neutral-0: #FFFFFF;
--p-neutral-100: #0B0D12;
TIER 02 — SEMANTIC

시맨틱 토큰

Semantic Tokens

목적을 가진 이름. 테마(Light/Dark)에 따라 다른 원시값을 참조합니다.

--sm-background-default
--sm-content-primary
--sm-border-subtle
TIER 03 — COMPONENT

컴포넌트 토큰

Component Tokens

특정 컴포넌트 전용. 시맨틱 토큰을 참조하며, 예외를 흡수합니다.

--cm-button-primary-bg
--cm-card-surface
--cm-input-border-focus
Why 3 tiers 하나의 예시: Primary 버튼의 배경이 다크 모드에서 조금 밝아져야 한다면 — 시맨틱 토큰(interactive-brand-default)을 바꾸면 전체 시스템이 흔들립니다. 대신 button-primary-bg라는 컴포넌트 토큰을 만들고 거기서 다른 원시값을 참조하게 합니다. 시스템은 흔들리지 않고, 예외는 정확히 격리됩니다.

토큰 네이밍 규칙 pattern

모든 토큰은 다음 패턴을 따릅니다. 왼쪽에서 오른쪽으로 갈수록 구체적입니다.

// Primitive --p-<category>-<scale> ex) --p-indigo-500 // Semantic --sm-<role>-<variant> ex) --sm-background-default ex) --sm-content-primary // Component --cm-<component>-<part>-<state> ex) --cm-button-primary-hover ex) --cm-input-border-focus
03·NAMING CONVENTIONS

이름은 계약이다

디자이너, 프론트엔드, 앱 개발자가 같은 컴포넌트를 다르게 부르면 시스템은 무너집니다. 이름은 팀 간의 계약입니다. 약어는 대부분 금지입니다.

권장 금지 이유
Button Btn 풀네임 원칙. 코드에서도 풀네임.
Navigation Nav 의미가 즉시 파악되어야 합니다.
BottomSheet BSheet / BtmSheet 두 단어 합성어는 PascalCase.
background-default bg-white 색 이름(white)이 아닌 용도(default)로.
content-primary text-dark 'dark'는 라이트 모드 기준의 말입니다.
interactive-brand-hover primary-hover 'primary'는 브랜드를 뜻하는지 모호.
size-400 spacing-16 값이 바뀌면 이름이 거짓말이 됩니다.
예외 업계 관례가 된 약어는 허용합니다: CTA, URL, SVG, FAQ. 단, 새로운 축약어를 만들지는 않습니다.
04·REFINEMENT

그라데이션 정책

v0.2에서 쓰인 두-색 그라데이션은 tacky했습니다. 확신 없는 톤 변화는 색의 의미를 흐리고, 브랜드의 일관성을 깎아먹습니다. v0.3부터는 솔리드가 기본입니다.

Before · After
v0.2 · 금지

두 색의 중간 지점이 탁해지고, 방향(135도)이 모든 요소에 기계적으로 적용되면서 개성 없는 인공적 톤이 됩니다. 색의 의미도 사라집니다 — 왜 보라에서 인디고로 변하는가?

v0.3 · 솔리드

각 색이 하나의 의미를 가집니다. 인디고는 브랜드, 초록은 성공, 호박은 강조, 빨강은 경고. 사용자가 색을 기호로 학습할 수 있습니다.

When gradients ARE allowed
그라데이션이 금지된 것은 아닙니다. 목적이 분명할 때만, 의도적으로 씁니다.
PROGRESS · 값의 변화를 나타냄
같은 색상군 안에서 명도만 변화 — 의도 명확
BRAND MOMENT · 한 번뿐인 순간
🎉 프리미엄 가입 완료
축하·성취의 특수 순간 한정
Rule 그라데이션을 쓰려면 두 가지 질문에 답해야 합니다. ① 왜 두 색이 필요한가? ② 이 상황이 반복되지 않는가? 둘 다 "예"가 아니면 솔리드로 갑니다.
05·UX WRITING

쓰여지는 것도 시스템이다

사용자는 버튼과 문구를 분리하지 않습니다. '저장'과 '저장하기'는 다른 컴포넌트입니다. 여기서 정의하는 원칙은 모든 화면의 모든 문장에 적용됩니다.

7가지 라이팅 원칙

PRINCIPLE 01

해요체로 일관되게

상황, 맥락, 화면 종류 불문 모든 문구는 해요체로 통일합니다. 합쇼체·반말·구어체 혼용은 브랜드 보이스를 깎아먹습니다.

저장되었습니다
저장했어요
PRINCIPLE 02

능동형, 사용자가 주어

수동형은 주체를 모호하게 만듭니다. 사용자가 주체인 문장이 행동을 명확히 합니다.

이메일이 전송되었습니다
이메일을 보냈어요
PRINCIPLE 03

부정 대신 긍정

"안 돼요"라고 말하지 말고, 무엇을 하면 되는지 알려줍니다. 사용자는 막힘이 아닌 길을 원합니다.

1,000원 이하는 송금할 수 없어요
1,000원부터 보낼 수 있어요
PRINCIPLE 04

잡초 뽑기 · 빼도 되는 단어는 뺀다

"혹시", "앞으로", "방금", "일단", "정말" — 넣어도 빠져도 의미가 같으면 뺍니다. 모바일 사용자는 읽지 않고 스캔합니다.

혹시 앞으로 알림을 받으시려면 설정해주세요
알림을 받으려면 설정해주세요
PRINCIPLE 05

버튼 라벨은 결과를 담는다

"확인"은 무엇을 확인하는지 모릅니다. 이 버튼을 누르면 무슨 일이 일어나는지 동사로 말합니다.

확인 / 완료 / OK
결제하기 / 보내기 / 신청하기
PRINCIPLE 06

에러는 막다른 길이 아니라 안내다

왜 발생했는지 간결하게, 어떻게 해결할지 구체적으로. "다시 시도해주세요"는 아무것도 알려주지 않습니다.

오류가 발생했어요. 다시 시도해주세요
네트워크가 불안정해요.
Wi-Fi를 확인해주세요
PRINCIPLE 07

숫자는 정확하게, 단위는 일관되게

"잠시", "많은", "곧"은 라이팅이 아니라 회피입니다. 가능한 한 구체적인 숫자로. 단위 표기(₩, 원, %, MB, 분)는 한 제품에 한 방식으로 통일합니다.

잠시 후 다시 확인해주세요 · 용량이 부족해요 · 곧 만료되는 쿠폰이 있어요
3초 뒤에 다시 확인해주세요 · 저장 공간이 500MB 남았어요 · 23일 뒤에 만료되는 쿠폰이 있어요

컴포넌트별 라이팅 규칙

자주 등장하는 컴포넌트는 문구도 패턴화합니다. 고민의 여지를 줄여 일관성을 높이는 방법입니다.

컴포넌트 금지 권장 이유
Dialog 왼쪽 버튼 취소 닫기 '취소'는 지금까지 한 작업이 취소될까 봐 사용자가 망설입니다.
Dialog 오른쪽 버튼 확인 · OK · 네 삭제하기 · 나가기 · 저장하기 결과를 담는 동사여야 무엇이 일어나는지 예측됩니다.
Empty State 제목 데이터가 없습니다 아직 저장한 항목이 없어요 어떤 데이터인지 구체적으로.
Empty State 액션 추가 첫 항목 추가하기 비어 있다는 상황을 맥락으로 가져갑니다.
Toast 성공 성공하였습니다 저장했어요 일반적 단어('성공') 대신 구체적 행위.
Loading 상태 Loading... · 잠시만요 불러오고 있어요 해요체 + 능동형.
Text Field 필수 표시 (필수) 빨간 별표 * 단어는 공간을 먹고, 기호는 국제적으로 통용됩니다.
Form 입력 안내 올바른 이메일을 입력해주세요 '@' 기호가 빠진 것 같아요 정확히 무엇이 문제인지 짚어줍니다.

에러 메시지 템플릿

모든 에러는 세 가지 질문에 답해야 합니다. 무슨 일이 일어났나 · 왜 그런가 · 어떻게 할까.

TEMPLATE A · 네트워크 오류
상황 + 원인 + 해결
연결에 실패했어요 네트워크가 불안정해요. Wi-Fi를 확인해주세요.
TEMPLATE B · 유효성 검증 실패
어디서 + 무엇이 틀렸나
휴대폰 번호를 다시 확인해주세요 11자리 숫자를 모두 입력해주세요.
TEMPLATE C · 시스템 점검
대상 + 시간 + 대안
하나은행 점검 중이에요 04월 21일 23시까지 송금이 어려워요. 다른 계좌를 써주세요.
Navigating Error 에러 메시지의 역할은 "문제를 알림"이 아니라 "다음 화면으로 넘어갈 수 있게 안내"입니다. 메시지가 끝나도 사용자는 한 발자국 더 갈 수 있어야 합니다.
